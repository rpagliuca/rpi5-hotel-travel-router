🧳 RPi5 Hotel Travel Router

Um Raspberry Pi 5 que conecta no Wi-Fi do hotel, cria seu Wi-Fi privado e roteia tudo por um exit node Tailscale. Configurável por um web app, sem SSH.

O que é

Você chega num hotel com Wi-Fi público e não-confiável. Em vez de conectar seus aparelhos direto nele, você liga o Pi: ele entra no Wi-Fi do hotel como cliente e cria seu próprio Wi-Fi privado. Seus dispositivos conectam nesse AP privado, e todo o tráfego sai criptografado por um exit node da sua Tailscale — como se você estivesse em casa.

[ Wi-Fi do hotel ] ↑ wlan0 (cliente / "STA" — uplink) [ Raspberry Pi 5 ] ← rádio único: AP e cliente no mesmo canal ↓ uap0 (AP privado — 192.168.88.1) [ laptop ] [ celular ] → tailscale0 → exit node → internet

Toda a configuração fica em config.yml (para o Ansible) e no uplink.conf do Pi (editado pelo web app). Nada pessoal vai para o repositório público.

Arquitetura

O Pi 5 tem um único rádio Wi-Fi. Ele faz dois papéis ao mesmo tempo, no mesmo canal (limitação física do modo concorrente):

  • STA / uplink (wlan0, wpa_supplicant): cliente do Wi-Fi do hotel.
  • AP (uap0, hostapd): a interface virtual do seu Wi-Fi privado, criada sobre a wlan0.

Fundação: o AP privado nunca cai 🛡️

A prioridade nº 1 é nunca perder acesso ao Pi. O AP privado é a âncora de segurança — é por ele que você entra no web app para consertar qualquer coisa.

  • Sempre sobe. No boot, o orquestrador tenta o uplink por ~40s e lê o canal dele; associando ou não, o hostapd sobe em seguida (no canal do uplink, ou no canal default se o uplink falhar). Wi-Fi do hotel fora de alcance/senha errada? O AP sobe assim mesmo.
  • Auto-recupera. hostapd com Restart=always e um watchdog a cada 60s que repara uap0, IP, hostapd e dnsmasq.
  • Senha pessoal. WPA2 com a sua senha — só você entra. SSH pela sub-rede do AP é sempre liberado.

A garantia

O início do AP não depende do sucesso do uplink nem do Tailscale. Se algo falhar, o AP continua no ar — e o web app com ele.

Modos: secure ↔ portal

Redes de hotel exigem login (captive portal) e são não-confiáveis. Daí dois modos, alternáveis pelo web app ou via CLI:

Instalação

Requisitos: Pi 5 com Raspberry Pi OS Bookworm ou Trixie (64-bit), sua chave SSH no Pi, e um exit node Tailscale já rodando na sua tailnet.

git clone https://github.com/rpagliuca/rpi5-hotel-travel-router
cd rpi5-hotel-travel-router/ansible
cp config.example.yml config.yml   # gitignored — segredos só aqui
$EDITOR config.yml                 # SSIDs, senha do AP, authkey, exit node
ansible-playbook playbook.yml -e @config.yml

Aplicando remotamente? Use o deadman

Os roles mexem na wlan0 e sobem o AP no mesmo rádio — se algo falhar você pode perder o acesso. Rode com -e deadman_rollback=true (veja Rede de segurança).

Todos os valores do seu ambiente ficam em config.yml (gitignored). Sem vault: como o arquivo nunca é comitado, os segredos jamais vão para o repositório — que é público e reaproveitável.

Web app — http://192.168.88.1

Conectado no seu AP privado, abra http://192.168.88.1 (⚠ http, não https) no navegador. Sem terminal, você:

  • Vê o status: uplink, internet, Tailscale + exit node, dispositivos no AP (auto-refresh).
  • Escolhe o Wi-Fi do hotel: escaneia, seleciona, aberta ou com senha.
  • Troca a banda (2.4 / 5 / auto) e o modo (secure / portal).
  • Manutenção: reiniciar ou fazer rollback.

Trocar de rede

O modelo é gravar e reiniciar: sua escolha vai para o uplink.conf (a fonte da verdade) e o Pi reinicia para subir na rede nova. Por quê reboot? No rádio único, o AP precisa estar no canal do uplink — e isso só pode ser fixado no boot, antes do AP subir (veja Aprendizados). A escolha persiste; um novo ansible-playbook não a sobrescreve.

Banda & scan

Escanear não precisa de toggle. O scan varre todas as bandas (2.4 e 5GHz) sempre — o toggle de banda só afeta a que o Pi vai conectar.

  • Ver redes 5GHz: é só escanear. (O rádio único sai do canal do AP por ~1s durante o scan — normal.)
  • Conectar numa rede 5GHz: primeiro troque a banda para 5GHz ou Auto; senão o filtro de 2.4GHz bloqueia a associação.
  • 5GHz usa só canais não-DFS (o AP não consegue seguir DFS). Default é 2.4GHz (mais compatível/estável).

Rede de segurança: deadman rollback 🪂

Aplicar via SSH remoto é arriscado. O deadman switch (opt-in com -e deadman_rollback=true) protege:

  • Arma antes de qualquer mudança de rede (primeira coisa do playbook). Se o apply morrer no meio, o rollback ainda acontece.
  • Enquanto valida, mantenha o heartbeat vivo (sudo travel-router-keepalive a cada < 5 min). Sem heartbeat → o Pi reverte ao estado base (wlan0 no NetworkManager, sem AP/NAT, Tailscale sem exit node) e reinicia.
  • Um perfil de fallback do uplink garante que o rollback sempre reconecta online.
sudo travel-router-keepalive   # mantém vivo enquanto valida
sudo travel-router-commit      # deu tudo certo → torna permanente
sudo travel-router-rollback    # reverte manualmente ao estado base

Aprendizados 🧠

Os problemas reais que moldaram o projeto — e as soluções.

1. Reiniciar o hostapd trava a firmware do rádio

No rádio único (brcmfmac), systemctl restart hostapd com o uplink ativo trava a firmware (-52) e derruba AP e uplink. Era o que a versão antiga fazia para "seguir o canal" do hotel — por isso o AP caía quando o hotel usava um canal diferente. E chan_switch (CSA) não é suportado nesse chip.

Solução: descobrir o canal do uplink antes do hostapd subir e fixá-lo no hostapd.conf. O AP sobe uma vez, no canal certo, e nunca reinicia. Trocar de canal = novo boot. Isso faz 5GHz funcionar sem dongle — o AP acompanha canal e banda.

2. A rede de segurança tem que armar cedo

O deadman como última etapa só protege quando tudo já deu certo — inútil se o apply morre no meio. Ele passou a ser a primeira coisa, antes de tocar na rede.

3. O rollback precisa de credencial de uplink

Um rollback devolveu a wlan0 ao NetworkManager sem nenhuma credencial → Pi órfão, offline permanente. Agora um perfil de fallback (com o SSID/senha do uplink) é instalado antes, garantindo que qualquer rollback volte online.

4. Cutover determinístico, não por "changed"

Handlers condicionados a mudança não disparavam num re-apply (config idêntica) e deixavam a wlan0 no limbo. O cutover do uplink virou um passo explícito e incondicional, com checkpoint que só segue quando o uplink tem IP.

5. Subir a STA destrói o AP virtual

(Re)inicializar o uplink no rádio único apaga a uap0. O orquestrador recria a uap0 depois que a STA sobe e antes do hostapd — de forma idempotente (cria só se faltar).

6. Repo público, zero segredo

Tudo específico do ambiente vive em config.yml e uplink.conf (gitignored). Sem ansible-vault (não faz sentido num repo reaproveitável). O README explica como criar o config.yml antes de aplicar.

Bônus: web app fora do ar? Provavelmente é https

Navegadores forçam https:// automaticamente; o app só serve http na porta 80. O firewall rejeita a 443 (com tcp reset) para o navegador cair de volta no http sozinho — mas se digitar, use http://.
